# 次フェーズ実装計画

## 現状

050番号はCloud Run上の日本語AI受付に接続済みです。通話ログは専用Firestore `speech-assistant-logs/callLogs` に保存され、Google Sheetsへ運用ビューとして追記されます。Sheetsの時刻は日本時間表示、電話番号は文字列扱い、エージェント発話の重複記録は修正済みです。

## 推奨する次フェーズ

次は「安全に運用し、管理画面で見られる状態」を優先します。React/RAG/WebRTCへ進む前に、通話ログと入口の安全性を固めます。

## Wave 2A: セキュリティと監査

対象Issue: #19

- Twilio webhook署名検証を `/incoming-call` に追加する。
- 不正なTwilio以外のリクエストを拒否する。
- 通話ログのPII項目、保持期間、閲覧権限を定義する。
- Cloud Runログに個人情報を出しすぎないよう、ログレベルを整理する。

完了条件:

- 正常なTwilio着信は通る。
- 署名不正リクエストは403で拒否される。
- FirestoreとSheetsへの保存は維持される。

## Wave 2B: 通話ログ品質

対象Issue: #19, #1, #10

- 実通話ログをもとにVAD設定を比較する。
- 顧客名、電話番号、希望日時の抽出失敗例を集める。
- 通話ログに「抽出信頼度」「要確認フラグ」を追加する。
- BigQuery移行を見据えた通話ログスキーマを固定する。

完了条件:

- SheetsとFirestoreで、運用者が折り返し要否を判断できる。
- 名前や電話番号が怪しい通話は「要確認」として残る。

## Wave 3: React管理画面

対象Issue: #12, #11

- Firestore `callLogs` の一覧を表示する。
- 通話詳細で文字起こし、要約、顧客名、電話番号、希望日時を表示する。
- ステータス、折り返し要否、要確認フラグで絞り込めるようにする。
- Realtimeイベント表示は、まず通話後ログから始める。

完了条件:

- Sheetsを開かなくても最新通話を確認できる。
- 管理画面から通話詳細と折り返し情報を確認できる。

## Wave 4詳細: Markdown/PDFナレッジCRUD

対象Issue: #30, #29, #14

- React管理画面でMarkdown本文を作成、編集、削除する。
- PDFをアップロードし、タイトル、カテゴリ、公開/下書き、更新者、更新日時を管理する。
- Cloud StorageまたはFirestoreに原本を保存する。
- PDFテキスト抽出、Markdown正規化、チャンク化、検索インデックス反映を行う。
- インデックス状態、失敗理由、再試行ボタンを管理画面に表示する。
- AIが参照したナレッジID、タイトル、ページ番号を通話ログに残す。

完了条件:

- 非エンジニアがUIからFAQや業務ルールを更新できる。
- 登録したMarkdown/PDFがAI応答で検索参照される。
- 参照元が通話ログと管理画面で確認できる。
